Authoritative Server Model

This project uses Colyseus to implement an authoritative server model. This is where the server is incharge of keeping track of the game state while the client job is only to show the end user state that the server is currently communicating. A strategy like this is very useful to prevent cheating and is commonplace in the industry. Due to the nature of the game and its turn based I saw no need to implement a client-side prediction model, since the lag between switching states would not cause much issues for the end user nor the game flow.

Command Pattern

After reading Game Programming Patterns by Robert Nystrom I learned about the “Command Pattern”. In this pattern one delegates tasks to be dealt with later, this is done so to decouple the actual algorithm with the code that surrounds it. This makes executing the messages received from the clients a lot easier for the server to handle. Rather than having a verbose file that contains all the functions of what needs to be triggered I can delegate them to their own object that can execute them.

Project Description

This game is a poker inspired card game. Player’s must battle their cards using a rock-paper-scissors type system. Battles will determine the draft order, the draft is when players choose their poker hand. After 5 rounds of battles and drafts the player with the highest poker hand wins.

What I learned

Throughout this project I gained a deeper understanding of an authoritative server model. Understanding how internet protocols work and using them to their advantage was very important to apply that knowledge within this project. Once I figured and created the server and client connection the rest of the game became a lot easier.
